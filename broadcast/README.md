# Broadcast

Challenge #3 из <https://fly.io/dist-sys/>. Реализация gossip-сервиса: ноды получают целочисленные значения от клиентов, обмениваются ими между собой так, чтобы каждая нода в итоге знала все значения, и отдают накопленный список по запросу.

Состоит из пяти стадий — 3a, 3b, 3c, 3d, 3e — с нарастающими требованиями (multi-node, partition tolerance, latency, message count). Один и тот же бинарь должен в идеале проходить все пять; на каждой стадии меняется только Maelstrom-команда.

## Протокол

Maelstrom присылает три типа сообщений:

| Запрос      | Поля                                          | Ответ          | Поля                  |
|-------------|-----------------------------------------------|----------------|-----------------------|
| `broadcast` | `message: <integer>`                          | `broadcast_ok` | —                     |
| `read`      | —                                             | `read_ok`      | `messages: [<int>...]`|
| `topology`  | `topology: {node_id: [neighbor_id, ...]}`     | `topology_ok`  | —                     |

- `broadcast` — «запомни число».
- `read` — «отдай все, что видел».
- `topology` — Maelstrom сообщает, кого считать соседями для gossip-а.

В сообщениях между нодами `src` начинается с `n` (`n0`, `n1`...), у клиентов — с `c` (`c1`, `c2`...).

## Стадии

### 3a — Single-node

**Условия:** одна нода, нет партишенов. Просто in-memory key-value store.

**Подход:** `HashSet<u64>` для значений, `HashMap<String, Vec<String>>` для топологии. На `broadcast` — insert + reply, на `read` — собрать `Vec` через `.iter().copied().collect()`, на `topology` — сохранить и ответить пустым ok.

**Команда:**
```bash
cargo build --release
./maelstrom/maelstrom test -w broadcast --bin ./target/release/broadcast \
  --node-count 1 --time-limit 20 --rate 10
```

**Статус:** ✅ пройдено (тег `stage-3a`).

### 3b — Multi-node, без партишенов

**Условия:** 5 нод, надёжная сеть. Maelstrom присылает `broadcast` на одну ноду, проверяет что значение видно на любой другой через `read`. Партишенов нет — доставка сообщений между нодами гарантирована (с возможной задержкой).

**Новая теория:** gossip / flooding. На `broadcast` нода:

1. Если число уже видела — игнор (защита от циклов).
2. Иначе — вставляет в `messages`, форвардит дальше соседям из топологии (кроме отправителя).

Дедупликация через `HashSet` гасит циклы — даже при полном графе сообщение быстро затухает.

**Что меняется в коде:**

- Кроме `into_reply` теперь нужно строить **инициативные** исходящие сообщения вручную (`Message { src, dst, body }`) — для рассылки соседям.
- В ветке `Broadcast` появляется ветвление: «это уже видели?» — и если нет, то цикл по соседям.
- `next_msg_id` инкрементируется чаще (на каждое исходящее, а не только на reply).
- Различение «от клиента или от ноды»: по `input.src.starts_with('c')` или по наличию в `self.topology`.

**Команда:**
```bash
./maelstrom/maelstrom test -w broadcast --bin ./target/release/broadcast \
  --node-count 5 --time-limit 20 --rate 10
```

**Статус:** 🚧 в работе.

### 3c — Fault-tolerant

**Условия:** 5 нод + `--nemesis partition`. Сеть может рваться и собираться. Maelstrom требует чтобы все значения, переданные клиентом, в итоге дошли до всех нод (eventual consistency).

**Что станет нужно:** retry / ack / persistent gossip. Простой «отправил и забыл» из 3b сломается под партишеном — сообщения теряются. Подходы:
- Каждое gossip-сообщение требует `broadcast_ok` от соседа. Если ack не пришёл за таймаут — повторить.
- Periodic anti-entropy: каждые N миллисекунд каждая нода отправляет соседям полный набор `messages` (или дельту). Соседи мерджат через `union`.

**Статус:** не начато.

### 3d — Efficient I

**Условия:** те же, что 3c, плюс budget на количество сообщений и latency. Maelstrom после прогона выведет статистику — нужно уложиться в лимиты.

**Что станет нужно:** оптимизация gossip-стратегии. Batching сообщений, выбор подмножества соседей (gossip-fanout вместо broadcast-to-all), spanning tree поверх случайной топологии.

**Статус:** не начато.
